# Demand as continuous market consumption, not a production ceiling

The original model treated national demand (100k m3/year across MDF, particle board, OSB) as a one-shot ceiling: once cumulative production crossed the target, `get_unmet_demands()` returned zero and all processors idled for the remainder of the 365-day run. Adding more waste types with transformation pathways caused the ceiling to be hit by day ~100, making the simulation useless for the remaining 265 days.

We replaced this with a market consumption process: a SimPy process that fires weekly, removes products from treatment operators' `product_to_sell` storage at a seasonally-modulated rate, and records fulfilled vs unfulfilled consumption. The annual demand envelope is the same 100k m3 — what changed is that it's consumed continuously over the year rather than checked as a cumulative ceiling.

This decision reshapes the entire control architecture:

- **PUSH operators** react to inventory drops caused by consumption (no explicit notification). Storage capacity is the implicit production target — no separate forecast parameter.
- **PULL operators** receive explicit consumption events via KanbanManager (`source_type="market"`), which cascade upstream to collectors and generators through the existing signal infrastructure.
- **Service level** becomes `total_consumed / total_attempted` computed from weekly consumption events — meaningful from day 1, not a binary hit/miss at year end.
- **Lost sales** are tagged by reason (`no_capability` vs `stockout`) to separate structural production gaps from operational failures.
- **Consumption is distributed** proportionally to each operator's processing capacity (set as `market_share` at construction time), uniform across all product types — operators that can't produce a product type see structural lost sales, which surfaces policy-relevant insights about regional capacity gaps.

## Considered options

- **Time-proportional demand**: `unmet = (target * fraction_of_year) - consumed`. Rejected because it conflates demand arrival with production triggering, bakes in linearity that makes seasonal extension painful, and creates edge cases (formula goes negative if system runs ahead) with no clean operational interpretation.
- **Per-product market share**: Only distribute consumption for products an operator can actually make. Rejected because it hides structural gaps that are policy-relevant for circular economy analysis (e.g., a region with no MDF capability has real unmet demand that suggests cross-regional transport or capacity investment).
- **Backlog instead of lost sales**: Unfulfilled consumption carries over. Rejected for now — lost sales is more realistic for commodity wood products (buyers don't wait), and it makes the service level metric harsher and more discriminating between strategies. Backlog is noted as a future research option.

## Consequences

- `_distribute_demand()` in SimulationManager deletes. Replaced by `market_share` parameter on each operator, set at construction.
- `SimulationState.get_unmet_demands()`, `check_all_demands_met()`, `demand_met_times` all delete.
- `processor.demand` as a persistent attribute deletes — neither set at init nor read meaningfully.
- `_apply_stock_strategy_to_demand_calculation()` deletes or refactors to operate purely on inventory levels.
- `product_storage` attribute on TreatmentOperator deletes. Under the old ceiling model it was a write-only overflow sink: production beyond `product_to_sell` capacity (itself rarely drained because `fulfill_demand` subtracted at production time) landed in `product_storage` with no drain path. With market consumption draining the finished-goods inventory continuously, the secondary buffer is incoherent. Mass balance simplifies to `products_made = products_consumed + finished_goods + production_discarded`. Monitoring (`waste_monitor.py: product_storage_by_type`) and visualization (`storage_visualization.py`) layers update to remove the secondary buffer track. Figure 14 (product-storage heatmap) disappears from results; methodology section should explicitly note the structural simplification rather than leave it unexplained.
- `product_to_sell` attribute and `product_to_sell_capacity` parameter rename to `finished_goods` and `finished_goods_capacity` across `TreatmentOperator` (`core/treatment.py`, which now also holds the former `core/treatment_utils.py` logic after the Candidate 1 inlining), `models/entities.py`, `core/facility_builder.py`, `monitoring/waste_monitor.py`, and `monitoring/visualization/storage_visualization.py`. The old name was meaningful only in contrast to the now-deleted `product_storage`; with one inventory remaining, the suffix is misleading. Aligns with standard SCM terminology — `waste_storage` is effectively raw materials, `finished_goods` completes the raw-materials-to-finished-goods pair familiar from inventory literature (and from the Tako & Robinson SCM-DES vocabulary used in the methodological framing). Done in the same pass as `product_storage` deletion to minimize churn.
- PUSH operators produce up to remaining `finished_goods` headroom, per output type. For each transformation, `amount_to_process = min(waste_storage[input], processing_capacity, headroom / efficiency)` where `headroom = finished_goods_capacity[output_type] - finished_goods[output_type]`. The clamp is per-output-type — a transformation producing OSB is throttled when OSB inventory is full even if MDF has headroom. When headroom is zero for an output, that transformation is skipped for the cycle while others may still proceed. No silent product discard; no idle headroom. Backpressure stays local — waste continues accumulating; if `waste_storage` overflows, the existing landfill mechanism (`handle_storage_event` → `WasteMonitor`) kicks in. PUSH does not propagate "inventory full" upstream as a signal (that would be PULL semantics). Mass balance: `products_made = products_consumed + finished_goods` with `production_discarded = 0` by construction. Rejected: full-skip semantics (would strand small headroom for full weekly ticks; the existing `min(waste_storage, processing_capacity)` clamp is already non-atomic, so adding a headroom clamp is the same kind of constraint, not an atomicity violation).
- `finished_goods_capacity = market_share × (annual_demand_total / 52) × 4 × 1.0` (4 weeks of buffer, no safety inflation). Initial inventory at simulation start = 2 weeks of buffer (50% of capacity), giving symmetric headroom for growth and depletion. Capacity is a candidate for sensitivity analysis — Monte Carlo over `weeks_of_buffer ∈ {2, 4, 6, 8}` to test whether PUSH/PULL contrast is stable; if so, robustness claim; if not, important context for results interpretation.
- Stock Strategy gates are waste-side only — they do not gate `finished_goods` production. In PUSH, production triggers on autonomous waste-state polling against the strategy threshold (continuing existing semantics); in PULL, production responds to every consumption event with event-volume (subject to the partial-batch clamp), and the strategy gates only the upstream waste-side signaling that fires when `waste_storage` drops below the threshold. Aligns with thesis Sections 5.3 and 5.4.3 ("proactive ordering logic, monitoring raw material inventory levels") — strategies are `(s, S)`-style replenishment policies on raw materials, not finished-goods buffer targets. PUSH and PULL stay distinguishable: same strategy parameter, different trigger semantics (state polling vs event reaction). Rejected: Option A (strategy targets `finished_goods` level) and Option C (gates both downstream and upstream) because (a) thesis doesn't have finished-goods buffer-target semantics, (b) under uniform 50% initial inventory the ON_DEMAND vs REORDER_50 contrast becomes degenerate at steady state, (c) extends current code's waste-side strategy gates without inventing a new finished-goods-side mechanism, (d) the bullwhip-quantification story (`var(orders)/var(consumption)` at each echelon, P2 in the metrics roadmap) maps cleanly onto `(s, S)` lot-for-lot vs threshold-triggered ordering — ON_DEMAND yields the textbook unit bullwhip ratio lower bound, REORDER_X yields amplified variance.
- Current PULL `_continuous_demand_signaling` (treatment.py:350-365) uses a fixed 10% threshold for upstream signaling, ignoring stock strategy entirely. The refactor routes strategy through this path: PULL-ON_DEMAND signals upstream every cycle waste was consumed, PULL-REORDER_X signals when waste_storage crosses X% threshold. Brings PULL behavior in line with PUSH `_should_trigger_collection_based_on_strategy` semantics.
- PULL consumption-event signal carries `attempted` volume (full demand including unfulfilled portion), not `consumed` volume. The PULL operator's production target = `attempted`, clamped by partial-batch headroom on `finished_goods` and by waste/efficiency availability. Demand-aware PULL: an emptied `finished_goods` does not death-spiral because the next tick's attempted volume continues to drive production. Bullwhip implication: PULL-ON_DEMAND yields `var(production_orders) = var(attempted_demand)` → unit bullwhip ratio at the treatment echelon (textbook lower bound for lot-for-lot ordering). Reviewer-defense: the demand-visible signal is the simulation analog of real-world JIT augmentations (CONWIP, minimum-WIP, periodic state reviews per Hopp & Spearman *Factory Physics*) — pure stateless kanban is a textbook abstraction, not a real production policy. Lost-sale visibility stays at the SimulationState/market level for reporting, not at the operator level for production decisions. Rejected: signal = consumed (death-spiral artifacts under spike scenarios; non-linear variance distortion muddies bullwhip interpretation), signal = (attempted, consumed) tuple (payload that nothing else uses).
- Signal-volume rule: `S = waste_storage_capacity` (order up to full capacity), applied uniformly across PUSH and PULL and all three strategy variants. When `waste_storage` drops below threshold `s`, the upstream signal carries `signal_volume = waste_storage_capacity - current_waste_storage`. ON_DEMAND becomes lot-for-lot by construction (s = S = capacity → signal_volume = amount-just-consumed). REORDER_X uses lot sizes proportional to `(1 - X) × capacity` per cycle — the textbook `(s, S)` formulation per Silver/Pyke/Peterson. Changes current PUSH `_get_reorder_quantity` behavior (lines 288-294 use S = threshold giving tight oscillation; baseline figures will change accordingly). `on_demand_buffer_ratio` and `on_demand_target_ratio` constants in `config/constants.py` become dead code. Bullwhip implication: REORDER_X under S = capacity creates measurable variance amplification at each echelon (lot-sizing effect), while ON_DEMAND yields the textbook unit bullwhip ratio lower bound — the contrast that grounds the methodological claim.
- PUSH/PULL contrast becomes more rigorous: "produce based on own inventory state" vs "produce in response to downstream consumption events."
- Service level becomes a harsher metric than the old ceiling model. PUSH strategies may perform relatively better than in prior results because their over-production cushion genuinely protects against stockouts in a way the old metric didn't reward. This shift in relative performance is itself a finding worth reporting.
- Mass balance assertion implemented as a new `monitoring/mass_balance.py: MassBalanceMonitor`.
  - **Scheduling.** A SimPy process scheduled at `t = 7, 14, 21, ...` (aligned with weekly consumption ticks) with priority lower than the consumption process so the monitor fires *after* all consumption events at that timestep have settled — avoids transient violations from mid-handoff intermediate states. API exposes `check_continuous()` (the scheduled hook) and `check_final()` (end-of-run, callable from test harnesses without scheduling overhead).
  - **Failure behavior.** Explicit `raise MassBalanceViolation(...)` with a custom exception class, NOT Python `assert` (stripped under `-O`). Configurable to warn+continue via flag — default raise for development, default warn for batch Monte Carlo runs (one bad seed shouldn't tank 99 good replications). Warnings emit to a structured log so post-hoc analysis can identify which seeds drifted. Mixed tolerance per `math.isclose` semantics: `abs(delta) <= max(rel_tol * max(abs(lhs), abs(rhs)), abs_tol)` with `rel_tol=1e-6`, `abs_tol=1e-9` — pure relative tolerance breaks near zero (early-run before generation begins, or for waste types inactive in some scenarios).
  - **Construction.** Accepts an `EntityRegistry` dataclass (`state`, `operators`, `generators`, `collectors`, `transport`) at construction rather than positional args — clean test seam (mock the registry), no constructor refactor when new entity types land.
  - **Counters as first-class state on SimulationState.** `processing_losses[W]` is an explicit counter incremented at the emission site (`_process_waste_transformation`: `processing_losses[W] += (1 - efficiency) × input_amount`). NOT derived from yield × processed at check time — derived quantities accumulate rounding error differently than direct counters and the invariant drifts for non-bug reasons.
  - **Telemetry on every check, not just on violation.** `check_continuous()` emits `(t, lhs, rhs, delta)` to a time-series file unconditionally. Float drift accumulating over a 365-day run is often visible long before it crosses tolerance; the time series catches it early.
  - **Tests required (highest-leverage part of the feature).** At least two synthetic unit tests in `tests/monitoring/test_mass_balance.py`:
    (1) construct registry + run a fake consumption event that violates the invariant (e.g., decrement `finished_goods` without incrementing `consumed`) → confirm `MassBalanceViolation` raised;
    (2) same setup with a valid event → confirm no raise. An untested invariant is worse than no invariant — it provides false confidence. Budget 30 minutes.
  - **Scope phasing: products in Phase A, waste deferred to a P2 follow-up.** Product invariant: `initial_finished_goods[P] + cumulative_produced[P] == cumulative_consumed[P] + current_finished_goods[P] + cumulative_discarded[P]` (`production_discarded` should be `0` by construction under partial-batch). No transit complications — products live only in `finished_goods`. Waste invariant requires non-trivial transit accounting (see next entry) and is deferred to a follow-up phase to avoid expanding Phase A scope. The monitor structure (`MassBalanceViolation`, `EntityRegistry`, `check_continuous`/`check_final`, tolerance, telemetry, tests) is reusable when the waste invariant lands.
- Transit accounting (deferred for waste-side mass balance, P2):
  - **Cross-region (collector → collector) is real and observable.** `collector.transfer_waste_to_region` (`core/collector.py:500`) decrements source storage immediately; `CollectorCompany._handle_completed_transport` (`core/collector.py`, inlined from the former `core/collector_utils.py` in Candidate 1) credits the destination on arrival. Volume lives in `collector.active_transports[].volume` between dispatch and arrival. Two-phase delivery: cross-region transport returns waste to the destination *collector*, not directly to the requesting treatment operator; treatment then pulls locally. Mass balance can read this state by summing `active_transports` across all collectors.
  - **Local pickup (generator → collector via vehicle dispatch) is real but not centrally observable.** `_dispatch_vehicle_for_collection` (`core/collector.py:143`) yields `env.timeout(travel_time)` twice (outbound + return). Between `_perform_collection_at_site` debiting the generator and `_add_to_collection_center` crediting the collector, waste is "in the vehicle" with no aggregated buffer. Each vehicle has scalar `current_load` but not per-waste-type, and there is no system-wide iterator over in-flight vehicles. Building the waste invariant requires (a) extending `Vehicle.current_load` to per-waste-type and (b) an iterator over outbound vehicles — both real refactor work outside Phase A scope.
- Two service-level metrics derived from the consumption event log: `full_service_level = total_consumed / total_attempted` (headline; includes both `no_capability` and `stockout` lost sales) and `operational_service_level = total_consumed / (total_attempted - no_capability_lost)` (diagnostic; measures policy effectiveness on operationally-fulfillable demand). Same accounting infrastructure (event log + reason tags) computes both; both available per-operator and per-product via filtering. Treat `full_service_level` as the primary across-policy comparator since the `no_capability` floor is identical across all six PUSH/PULL × strategy configurations. Treat `operational_service_level` as the within-feasibility diagnostic. Rejected: single harsh metric (hides the diagnostic split between operational and structural gaps that the reason-tagging was introduced to enable), feasible-only (invites reviewer pushback on definition-shopping).
