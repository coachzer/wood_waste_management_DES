# Architecture deepening, sequenced around the demand refactor

An architecture review (2026-05-22, archived as `Architecture review — wood_waste_management_DES.htm`; captured in `.scratch/architecture-deepening/PRD.md`) applied Ousterhout's *A Philosophy of Software Design* to the codebase (~5,600 LOC, 35 files) and found six structural weaknesses: three shallow `*_utils.py` modules that displace complexity rather than reduce it, a `SimulationState` singleton coupling 20+ call sites to global mutable state, stock-strategy logic scattered across 5 locations with duplicated thresholds, two parallel entity hierarchies requiring manual field mapping, five thin initialization modules, and pure decision logic embedded inside SimPy generators that cannot be tested without a full simulation. None are bugs — they are testability and extensibility costs.

The complication is that the other large open work item — the demand-model refactor (ADR 0002) — rewrites many of the same files (`state.py`, `treatment.py`, `simulation_manager.py`) and deliberately changes behavior. So the decision is not just *what* to refactor but *in what order relative to the demand refactor*, because the wrong order means doing the same work twice.

We will deepen the architecture in a hybrid order: do the one candidate that genuinely helps the demand refactor first, then the demand refactor, then the candidate that the demand refactor would otherwise force us to redo.

## Considered options

- **Full architecture first (all of Phase 1, then demand refactor).** Cleanest codebase going into the refactor. Rejected because Candidate 3 (strategy extraction) consolidates the exact stock-strategy logic the demand refactor rewrites — signal-volume becomes `S = capacity`, the `on_demand_buffer_ratio` / `on_demand_target_ratio` constants become dead code, PULL signaling moves from a fixed 10% threshold to strategy-aware. Extracting strategies into protocol classes against current semantics, then immediately rewriting those classes, is wasted effort.
- **Demand refactor first, architecture later.** Maximizes time-to-research-value (the demand refactor is the methodological contribution that feeds the papers; architecture is infrastructure that produces no new figures). Rejected as the sole plan because the demand refactor is large and behavior-changing, and is materially easier to implement and debug on the inlined single-module codebase that Candidate 1 produces. We keep its spirit — research value stays the priority — by doing only the cheap, helpful refactor first.
- **Hybrid (chosen).** Candidate 1 (inline utils) now → demand refactor → Candidate 3 (strategy modules) after, with its final semantics. Candidate 2 (dependency injection) is optional-before, slotted in only if the demand refactor's mass-balance tests want the no-singleton test seam early. Candidates 4–6 are evaluated after Phase 1, per the review.

## Key decisions within this design

- **Candidate 1 (inline `*_utils.py`) is the immediate next step.** Inline `generator_utils.py`, `collector_utils.py`, `treatment_utils.py` back as methods on `WasteGenerator`, `CollectorCompany`, `TreatmentOperator`. This is pure locality — it survives any later behavior change, it shrinks the demand refactor's surface (it edits one file per entity, not two — `treatment_utils.py::fulfill_demand` is touched by the refactor anyway), and it is verifiable against the current git baseline before that baseline shifts. Accepted tradeoff: `treatment.py` grows to ~1030 LOC; the review endorses this (locality beats file size for a single deep module).
- **Candidate 3 (strategy/policy modules) is deferred until after the demand refactor.** Extract `StockStrategy` and `InventoryPolicy` as protocol classes once, against the post-refactor semantics, so the consolidation happens against the rules that will actually ship.
- **Candidate 2 (replace the singleton with injected state) is optional-before.** It buys the test seam that the demand refactor's mass-balance tests want (no `_instance = None` reset ritual). It changes how state is wired, which is a different axis from the demand refactor's rewrite of what state *contains*, so the rebase is modest. Sequence it before the demand refactor only if the test seam is wanted early; otherwise after.
- **Candidates 4 (merge entity hierarchies), 5 (deepen FacilityBuilder), 6 (separate SimPy from decisions) stay deferred.** Re-evaluate after Phase 1, per the review — Candidate 6 especially may become unnecessary once Candidate 3 extracts the strategy patterns.
- **The validation gate is behavior preservation.** Each architecture candidate must leave simulation outputs identical: run `main.py --mode baseline --replications 10`, compare KPIs (total waste, costs, demand satisfaction) within floating-point tolerance, no new tracebacks. This gate is only meaningful against the *current* baseline, which is another reason to land the behavior-preserving refactors that go before the demand refactor while that baseline still holds.

## Consequences

- Candidate 1 deletes three files (~488 LOC of shallow wrappers) and updates their call sites (e.g. `generator.py` → `handle_overflow`, `collector.py` → completed-transport handling). No architectural redesign — functions move and their signatures collapse into `self.` access.
- The demand refactor (ADR 0002) begins on the inlined codebase, so its file-surface pointers in `.scratch/NEXT-SESSION-demand-refactor.md` that reference `*_utils.py` resolve to the owning modules.
- `.scratch/architecture-deepening/PRD.md` remains the operational issue list; this ADR records the decision and the sequencing rationale. The review HTML is the source artifact.
